# Homework_12-08

## Задание 1. Резервное копирование

Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

### 1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

### 1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

Приведите ответ в свободной форме.

### Решение:

1.1. Восстановление данных за предыдущий день в полном объёме

Для восстановления данных в полном объёме за предыдущий день больше всего подходит дифференциальный бэкап (differential backup).

Этот вид резервного копирования копирует только новые и изменённые данные с момента последнего полного бэкапа, делая его более быстрым и экономным по сравнению с ежедневным полным. Для восстановления вам потребуется только два файла: последний полный бэкап и самый последний дифференциальный бэкап, что значительно ускоряет процесс, так как не требует последовательного применения множества промежуточных копий.

1.2. Восстановление данных за час до предполагаемой поломки

Для такой задачи идеально подходит инкрементное резервное копирование (incremental backup) в связке с журналами транзакций (WAL).

Инкрементный бэкап сам по себе копирует только изменения с момента последней любой операции (полной или инкрементной), позволяя делать очень частые копии и минимизируя потерю данных. Однако для восстановления на момент времени, отличный от времени создания бэкапа (например, час до поломки), применяется технология Point-in-Time Recovery (PITR) в PostgreSQL или бинарные логи в MySQL. Суть PITR в том, что вы восстанавливаете базу данных из полной резервной копии, а затем «накатываете» на неё все журналы транзакций, которые были созданы между моментом создания полной копии и заданным часом. Таким образом, вы можете восстановить базу данных в состояние, которое было на 1 час раньше, «откатив» последующие изменения.

## Задание 2. PostgreSQL

### 2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

Приведите ответ в свободной форме.

### Решение:

Пример команды резервного копирования (pg_dump)

Это полный логический бэкап базы данных, который создает SQL-скрипт для её воссоздания. Согласно документации, простейший синтаксис:
bash

pg_dump имя_базы_данных > файл_дампа.sql

Пример с параметрами подключения к удалённому хосту:
bash

pg_dump -h localhost -p 5432 -U username -d database_name > backup.sql

    -h — хост сервера БД

    -p — порт для подключения

    -U — имя пользователя для подключения

    -d — имя базы данных, которую нужно сохранить. Если не указывать, то по умолчанию используется имя пользователя

Пример команды восстановления (pg_restore)

Важно: pg_restore используется для восстановления из архивных форматов (-Fc (custom) или -Fd (directory)), а не из простого SQL-скрипта (для последнего используется утилита psql).

Пример команды для восстановления из архивного файла:
bash

pg_restore -U postgres -d database_name backup_file.dump

    -U — имя пользователя для подключения

    -d — целевая база данных, в которую необходимо восстановить содержимое. Перед восстановлением эту базу необходимо создать заранее

    backup_file.dump — путь к архивному файлу с резервной копией

## Задание 3. MySQL

### 3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

Приведите ответ в свободной форме.

### Решение:

Инкрементное резервное копирование в MySQL можно выполнить с помощью проприетарной утилиты mysqlbackup от Oracle (входит в MySQL Enterprise Backup) или с помощью бесплатной утилиты Percona XtraBackup. Команды выполняются после создания базового полного бэкапа.

Для выполнения инкрементного бэкапа используется опция --incremental или --incremental-with-redo-log-only.

Пример команды для создания инкрементного бэкапа:
bash

mysqlbackup --defaults-file=/etc/my.cnf -uroot --backup-dir=/path/to/incremental_backup_dir --incremental --incremental-base=history:last_full_backup backup

    --defaults-file — путь к конфигурационному файлу MySQL

    --backup-dir — директория, куда будет сохранён инкрементный бэкап

    --incremental — ключевая опция, указывающая, что бэкап должен быть инкрементным

    --incremental-base — указывает, на основе какого предыдущего бэкапа (полного или инкрементного) создаётся новый

    backup — команда, выполняющая резервное копирование
